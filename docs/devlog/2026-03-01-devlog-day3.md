## Technical Investigation: Call Audio Stack

### Starting Point

`callaudiod` was logging `no available output found! no available input found!` on every call. Calls connected but no audio routed through the earpiece.

### Investigation Path

**Step 1 — PipeWire has no audio devices**

`wpctl status` showed a completely empty Audio section — no devices, sinks, or sources. Only video/camera devices visible. WirePlumber logs showed no errors, just silence.

**Step 2 — ALSA sees the hardware fine**

`aplay -l` showed the OnePlus 6T sound card (`card 0: O6T`) with 7 PCM devices including `VoiceMMode1`. UCM profiles exist at `/usr/share/alsa/ucm2/OnePlus/fajita/`.

**Step 3 — Two audio systems fighting**

The image runs both PulseAudio and PipeWire. PulseAudio owns the audio hardware via `postmarketos-base-ui-audio-backend-pulseaudio`. PipeWire handles video/camera only. callaudiod uses the PulseAudio backend (`callaudiod-pulse`) — so PipeWire having no audio devices was irrelevant.

**Step 4 — PulseAudio has the Voice Call profile**

`pactl list cards` revealed the smoking gun:

```
Profiles:
  Voice Call: Make a phone call (sinks: 1, sources: 1, available: yes)
  HiFi: HiFi quality Music. (sinks: 2, sources: 3, available: yes)
Active Profile: HiFi
```

The Voice Call UCM profile exists, works, and exposes `[Out] Earpiece` and `[In] Mic`. But PulseAudio stays on HiFi — callaudiod is not switching it.

**Step 5 — Manual profile switch confirms fix**

```bash
pactl set-card-profile alsa_card.platform-sound "Voice Call"
```

Immediate result: `Voice_Call__Earpiece__sink` appears. Test call — clear audio, no echo.

**Step 6 — Why callaudiod fails**

callaudiod logs `no available output found` because it queries PulseAudio for a voice call sink *after* the call has already ended — the profile has already switched back to HiFi by the time it checks, or it never switched at all. The ModemManager integration in callaudiod is not functioning correctly on this image.

### Root Cause

callaudiod is not automatically switching PulseAudio to the `Voice Call` profile when a ModemManager call becomes active. The profile exists and works — the automation layer is broken.

---

## The Fix: mahala-call-audio.sh

A shell script watches ModemManager DBus `StateChanged` signals and switches the PulseAudio profile accordingly.

**DBus signal structure (captured via dbus-monitor):**
```
interface=org.freedesktop.ModemManager1.Call member=StateChanged
  int32 <old_state>
  int32 <new_state>
  uint32 <reason>
```

State values:
- `4` = active (call connected)
- `7` = terminated (call ended)

**Script:** `/usr/local/bin/mahala-call-audio.sh`

```bash
#!/bin/sh
dbus-monitor --system "type='signal',interface='org.freedesktop.ModemManager1.Call',member='StateChanged'" 2>/dev/null | \
while read -r line; do
    case "$line" in
        *"interface=org.freedesktop.ModemManager1.Call"*)
            read -r old_line
            read -r new_line
            read -r reason_line
            new_state=$(echo "$new_line" | grep -o '[0-9]*')
            if [ "$new_state" = "4" ]; then
                pactl set-card-profile alsa_card.platform-sound "Voice Call"
            elif [ "$new_state" = "7" ] || [ "$new_state" = "6" ]; then
                pactl set-card-profile alsa_card.platform-sound "HiFi"
            fi
            ;;
    esac
done
```

**Service:** `~/.config/systemd/user/mahala-call-audio.service`

Enabled as a user systemd service, starts automatically on login, restarts on failure.

### Result

- Profile switches to `Voice Call` automatically when call becomes active
- Switches back to `HiFi` when call terminates
- Clear audio both directions
- No echo or feedback
- Confirmed with `watch -n 1 "pactl list sinks short"` showing real-time profile switching
